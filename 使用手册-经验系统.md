# 经验系统使用手册

**版本:** 1.0
**更新:** 2026-06-16

---

## 一、这个系统做什么

一句话：**你用 Claude Code 写代码，遇到 bug、做技术决策、踩坑填坑，系统在后台自动把你的经验存起来。下次碰到类似问题，Claude 会自己翻出来提醒你。**

全程你不需要做任何额外操作。

---

## 二、你需要安装什么

只需要 **llm_wiki 应用**（包含知识库 + Hook 脚本 + MCP Server + 经验提取工具，全部在一个安装包里）。

| 组件 | 位置 |
|------|------|
| 知识库应用（存储 + 搜索 + Ingest） | llm_wiki 安装目录 |
| Hook 脚本 | `<安装目录>/experience-system/tools/capture_session.py` |
| MCP Server | `<安装目录>/mcp-server/dist/src/index.js` |
| 经验提取 | `<安装目录>/tools/extract_experiences.py` |

**前提：**
- llm_wiki 应用**正在运行**（否则 Source Watch 检测不到新文件，经验不会自动入库）
- MCP Server 可用（用于会话中搜索经验）

## 三、场景一：新建一个项目，开启经验积累

### Step 1：在 llm_wiki 中创建一个 wiki 项目

打开 llm_wiki 应用 → 新建项目 → 选择 **Experience（🧠）** 模板。

记下项目的路径，比如 `D:\work\my-experiences`。

这个项目的 `schema.md` 只定义了 6 种经验类型（bug/decision/howto/agent-error/pattern/template），不会有 source/entity/concept 那些通用知识库的东西。

### Step 2：配置开发项目的 `.claude/settings.json`

**去你平时写代码的项目根目录**（不是 wiki 项目），创建或编辑 `.claude/settings.json`：

```json
{
  "hooks": {
    "SessionEnd": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "python D:/work/llm_wiki/experience-system/tools/capture_session.py --stdin"
      }]
    }]
  },
  "env": {
    "LLM_WIKI_PROJECT": "smart-lock-firmware",
    "LLM_WIKI_DOMAIN": "embedded-arm",
    "LLM_WIKI_ROOT": "D:/work/llm_wiki",
    "LLM_WIKI_OUTPUT_DIR": "D:/work/my-experiences/raw/sources/experiences"
  }
}
```

> **注意：** Hook 使用 `matcher` + `hooks` 嵌套结构。`matcher` 为空字符串表示匹配所有操作（SessionEnd 是生命周期事件，不限定特定工具）。

**四个关键配置：**

| 配置 | 填什么 | 示例 |
|------|--------|------|
| `LLM_WIKI_PROJECT` | 项目名，经验页面会标注来自哪个项目 | `smart-lock-firmware` |
| `LLM_WIKI_DOMAIN` | 技术领域，用于跨项目归类 | `embedded-arm`、`web-frontend`、`python-backend` |
| `LLM_WIKI_ROOT` | llm_wiki 安装目录 | `D:/work/llm_wiki` |
| `LLM_WIKI_OUTPUT_DIR` | 经验文件输出到哪个 wiki 项目 | `D:/work/my-experiences/raw/sources/experiences` |

> **Linux 用户注意：** Ubuntu 上 Python 命令是 `python3` 而非 `python`。路径用正斜杠 `/`。详见 [部署指南.md](部署指南.md)。

### Step 3（可选）：配置 CLAUDE.md + Skill，让 LLM 主动搜索经验

上一步的 `.claude/settings.json` 只负责**会话结束时自动提取经验**。如果你想让 LLM 在**开发过程中**主动搜索知识库、碰到错误时提醒你，再配置两个文件：

**3a. `CLAUDE.md` — 行为规则**

```markdown
## 经验系统

本项目配置了经验积累。会话结束时经验会自动提取。

### 知识库位置
Wiki 项目路径 + 通过 MCP `llm_wiki_search` 搜索。

### 会话启动
启动时通过 MCP 搜索本项目的相关经验，告知用户。

### 自动搜索
遇到以下情况时，主动通过 MCP `llm_wiki_search` 搜索知识库：
- 编译/运行时错误（含 error、failed、panic、cannot、undefined is not 等）
- 配置文件不生效、格式错误
- 环境依赖问题（找不到工具、版本不兼容）
- 重复性工作（用户说"上次怎么做的"、"又遇到这个"）

搜索到相关经验时，告知用户：⚠️ 知识库有相关经验：[[PageName]] — 摘要
无匹配时继续正常排错，问题解决后确认是否产生新经验。

### 手动标记经验
- xp — 由 Skill 拦截，只输出轻量标记，不写文件
- "提炼经验" / "记录这个坑" — 同样只做标记
```

**3b. `.claude/skillsxp.md` — 轻量标记 Skill**

创建目录和文件：

```
.claude/skillsxp.md
```

内容从 llm_wiki 安装目录下的 `experience-system/skillsxp.md` 复制。

> **为什么需要 Skill？** 没有 Skill 时 `xp` 是纯文字规则，LLM 可能当场生成页面、写文件、更新索引——这会消耗大量上下文，打断开发思路。有了 Skill，系统级拦截 `xp`，只输出几行轻量标记，实际的页面生成交给 SessionEnd 后的 Ingest 管线。
>
> **不加会怎样？** SessionEnd Hook 照样自动提取经验。只是手动标记时可能消耗较多上下文。

### Step 4：正常使用

配置完就可以忘了它。以后每次你在这个项目里用 Claude Code，会话结束时经验自动提取。

## 四、场景二：继续一个已有项目

如果你在一个已经配过经验系统的项目里继续工作：

1. **什么都不用做。** `.claude/settings.json` 已经在项目里，Hook 自动生效。
2. 如果换了电脑或重新 clone 了项目，确认 `.claude/settings.json` 存在即可。

## 五、场景三：多项目共享一个经验库

如果你有多个开发项目（比如 `project-a` 和 `project-b`），想把经验都存在同一个 wiki 项目里：

每个开发项目都配置 `.claude/settings.json`，但：
- `LLM_WIKI_PROJECT` 填各自的项目名
- `LLM_WIKI_OUTPUT_DIR` 指向**同一个** wiki 项目

```json
// project-a/.claude/settings.json
"LLM_WIKI_PROJECT": "project-a",
"LLM_WIKI_DOMAIN": "web-frontend",
"LLM_WIKI_OUTPUT_DIR": "D:/work/my-experiences/raw/sources/experiences"

// project-b/.claude/settings.json
"LLM_WIKI_PROJECT": "project-b",
"LLM_WIKI_DOMAIN": "embedded-arm",
"LLM_WIKI_OUTPUT_DIR": "D:/work/my-experiences/raw/sources/experiences"
```

这样两个项目的经验都汇入同一个知识库，靠 `project` 和 `domain` 字段区分来源。

## 六、日常使用：会发生什么

### 会话启动时

如果项目配置了 `LLM_WIKI_PROJECT`，Claude 会：
1. 搜索知识库中该项目相关的经验
2. 告诉你"本项目有 N 条相关经验"
3. 在后续工作中自动参考这些经验

### 会话进行中

当你遇到以下情况时，Claude 会主动搜索知识库：
- 编译/运行时错误
- 配置文件不生效
- 重复性工作（"上次怎么做的"）

### 会话结束时

Hook 自动运行：
```
transcript JSONL
  → 过滤噪音
  → 提取文本
  → 写 Markdown（带 project/domain frontmatter）
  → 放入 raw/sources/experiences/
  → llm_wiki 检测到新文件
  → 自动 Ingest（LLM 分析 → 生成 bug/decision/howto 等页面）
  → 向量嵌入 → 可搜索
```

### 手动提取经验

如果你觉得某件事特别值得记录，可以直接在对话中说：
- `xp`
- "提炼经验"
- "记录这个坑"

Claude 会从当前会话提取经验并写入 wiki。

## 七、查找经验

### 方式一：对话中自动搜索

Claude 碰到错误时会自己搜。你也可以直接问："知识库里有没有关于 X 的经验？"

### 方式二：在 llm_wiki 应用中浏览

打开 wiki 项目，按类型浏览：
- `wiki/bugs/` — 缺陷与修复
- `wiki/decisions/` — 技术决策
- `wiki/howto/` — 操作步骤
- `wiki/agent-errors/` — Claude 犯过的错
- `wiki/patterns/` — 重复出现的问题模式
- `wiki/templates/` — 可复用的检查清单

### 方式三：Obsidian 打开

wiki 项目就是普通的 Markdown 文件，可以用 Obsidian 打开查看图谱。

## 八、经验页面结构

所有经验页面有统一的 frontmatter：

```yaml
---
type: bug                          # bug|decision|howto|agent-error|pattern|template
title: HardFault 因 _estack 符号冲突
tags: [hardfault, startup, linker] # 3-5 个技术标签
related: [pattern-linker-startup]  # 交叉引用其他页面
project: "smart-lock-firmware"     # 来源项目
domain: "embedded-arm"             # 技术领域
sources: ["experiences/xxx.md"]    # 来源 transcript
created: 2026-06-16
updated: 2026-06-16
---
```

每种类型的 body 结构不同，但核心思路一致：**把坑写清楚，让下次能直接用。**

## 九、常见问题

### Q: 我不想某个项目的经验被自动提取

删除那个项目的 `.claude/settings.json` 中的 SessionEnd Hook 即可。

### Q: 换个电脑怎么继续用

把 `.claude/settings.json` 提交到 git（它在 `.claude/` 下，不会被自动忽略）。换电脑后确认 llm_wiki 和 llm-wiki-agent 的路径一致，或调整 `command` 中的路径。

### Q: 经验质量不好怎么办

In ingest 管线有三层校验：
1. Prompt 引导 LLM 按结构生成
2. Schema 约束只接受 6 种经验类型（不是经验类型的会被拒绝）
3. Page merge 做去重保护

如果某条具体经验质量差，可以在 llm_wiki 中手动编辑或删除。

### Q: 会产生重复经验吗

不会。如果新提取的经验与已有页面相似，page-merge 会跳过或合并，不会创建重复页面。

### Q: 跨项目的模式挖掘（Phase 3）

尚未实现。当经验积累足够多（≥10 条 bug 页面）后可以开启。
